# 项目开发文档

## 1. 项目概述

### 1.1 项目名称

美团智能决策助手交互原型

### 1.2 项目目标

本项目用于将高保真原型稿还原为可交互的前端 Demo，重点体现三类本地生活决策场景中的页面结构、筛选方式、状态切换和交互反馈。

项目覆盖三条业务线：

- 外卖
- 到店餐饮
- 酒店

### 1.3 已实现范围

当前版本已经实现以下页面链路：

1. 首页入口
2. 候选选择
3. 候选池
4. 智能对比
5. 评价风险
6. AI 总结
7. 确认选择
8. 详情页

当前版本已经实现以下交互能力：

- 业务线切换
- 页面状态切换
- 搜索词输入
- 候选对象选择与移除
- 排序筛选切换
- 智能对比维度切换
- 弹窗提示
- 底部分享面板
- 页面切换与局部动画

## 2. 开发环境搭建

### 2.1 推荐环境

- 操作系统：Windows 10 / Windows 11
- 编辑器：VS Code
- Node.js：18 及以上
- 包管理器：npm
- 版本管理：Git

说明：

- 项目当前脚本为 PowerShell 版本，因此 Windows 环境最适合直接运行。
- 如果在 macOS 或 Linux 环境中开发，需要补充对应的 shell 脚本。

### 2.2 打开项目

建议直接使用 VS Code 打开整个项目根目录：

```text
C:\Users\Wheel\Documents\美团app\prototype-app
```

不要只打开单个文件，否则不方便查看完整目录结构和运行脚本。

### 2.3 安装依赖

在项目根目录执行：

```bash
npm install
```

安装完成后，项目会具备以下运行依赖：

- React
- ReactDOM
- Framer Motion
- TypeScript
- Tailwind CSS
- PostCSS
- Vite 相关依赖

### 2.4 启动前检查

建议先执行类型检查：

```bash
npm run check
```

如果没有报错，再继续构建和预览。

## 3. 技术栈说明

### 3.1 核心技术

- React：负责组件化结构和交互逻辑
- TypeScript：负责类型约束和数据结构定义
- Tailwind CSS：负责原型页面的样式快速搭建
- Framer Motion：负责页面切换、面板弹出、局部动效

### 3.2 工程脚本

- `scripts/check.ps1`：执行类型检查
- `scripts/build.ps1`：生成 CSS 和 JavaScript 构建产物
- `scripts/dev-server.ps1`：启动本地预览服务
- `scripts/dev-server.mjs`：提供静态 HTTP 服务
- `scripts/validate.mjs`：检查关键文件是否齐全

### 3.3 当前运行模式

本项目不是标准的前后端分离线上项目，而是更适合课程展示和原型演示的静态交互 Demo。

当前采用的运行方式是：

1. 用 Tailwind CLI 生成样式
2. 用 TypeScript 编译 `src/` 到 `dist/`
3. 用静态服务器从项目根目录提供访问

## 5. 核心代码结构说明

### 5.1 入口文件

#### `index.html`

职责：

- 提供页面基础 HTML 结构
- 挂载 `#root`
- 引入构建后的样式文件 `dist/styles.css`
- 引入 React、ReactDOM、Framer Motion 的运行时脚本
- 加载构建后的 `dist/main.js`

#### `src/main.tsx`

职责：

- 作为 React 应用入口
- 将 `App` 挂载到 `#root`

### 5.2 页面主控文件

#### `src/App.tsx`

这是项目最核心的文件，主要负责：

- 维护当前业务线状态
- 维护当前页面状态
- 维护搜索词、排序方式、候选对象
- 组织页面跳转关系
- 控制弹窗和底部分享面板
- 将数据与组件组合为完整交互流程

可以把它理解为“整个原型的页面流程调度中心”。

### 5.3 数据配置文件

#### `src/data/prototypeData.ts`

这是项目的数据中心，主要负责：

- 定义业务类型
- 定义页面类型
- 定义排序类型
- 定义候选对象的数据结构
- 定义三条业务线的页面文案、对比字段、筛选项和候选对象数据

这种写法的优点是：

- 页面结构与数据内容解耦
- 修改文案、排序项、候选对象时，不需要大量改动页面组件
- 三条业务线可以复用同一套页面逻辑

### 5.4 组件目录

#### `src/components/mds/`

这是项目的基础组件层，主要包含：

- `Button.tsx`：按钮组件
- `BottomSheet.tsx`：底部弹出面板
- `Modal.tsx`：居中弹窗
- `PhoneShell.tsx`：手机壳与状态栏容器
- `SearchField.tsx`：搜索框
- `SegmentedControl.tsx`：分段切换控件
- `Surface.tsx`：卡片容器

这些组件的作用是：

- 统一 UI 风格
- 降低页面重复代码
- 方便后续继续扩展或替换视觉细节

### 5.5 依赖包装层

#### `src/vendor/`

这一层主要负责从全局运行时中取出：

- React
- ReactDOM
- Framer Motion

这样做的原因是：

- 当前版本的 `index.html` 先通过脚本把这些库挂到全局
- 业务代码再通过 `vendor` 层进行统一引用
- 有利于保持源码结构清晰

## 6. 页面与交互实现说明

### 6.1 页面流程

当前项目的主要流程如下：

```text
首页
  -> 候选选择
  -> 候选池
  -> 智能对比
  -> 评价风险
  -> AI总结
  -> 确认选择
  -> 详情页
```

### 6.2 首页

功能：

- 展示三条业务线入口
- 展示原型说明
- 支持从左侧控制区切换页面状态

### 6.3 候选选择页

功能：

- 搜索候选对象
- 切换筛选排序方式
- 勾选或取消候选对象
- 进入候选池

说明：

- 当前三条业务线的候选选择页都采用统一结构
- 不同业务线只替换文案、候选数据和字段内容

### 6.4 候选池页

功能：

- 查看当前已加入对比的对象
- 移除对象
- 满足数量条件后进入智能对比

### 6.5 智能对比页

功能：

- 展示对比对象的多维字段
- 支持切换不同对比维度
- 进入价格更新、风险分析、AI 总结等后续步骤

### 6.6 评价风险页

功能：

- 展示高频好评与差评关键词
- 提示潜在风险和适用人群

### 6.7 AI 总结页

功能：

- 按推荐优先级展示结论
- 给出推荐理由、可能风险和适合人群
- 支持收藏、分享、确认下单和更换对象

### 6.8 确认选择页

功能：

- 汇总关键决策字段
- 提供下单或收藏动作

### 6.9 详情页

功能：

- 展示对象详情
- 展示推荐内容或房型内容
- 提示使用说明

## 7. 状态管理说明

本项目当前没有引入额外状态管理库，而是使用 React 自带能力完成状态组织。

主要状态包括：

- 当前业务线
- 当前页面
- 当前排序方式
- 当前搜索词
- 当前已选对象
- 当前对比标签页
- 当前弹窗状态

这种方式适合当前原型项目，优点是：

- 结构简单
- 可读性高
- 修改成本低

如果后续页面继续变多，或者接入真实接口，可以再考虑拆分状态层。

## 8. 样式与动画说明

### 8.1 样式体系

项目样式主要由两部分组成：

1. `src/index.css`
   - 全局样式
   - 背景与滚动条样式
   - 字体与基础 reset
2. Tailwind CSS
   - 负责页面布局、色彩、圆角、阴影、间距等原子类

### 8.2 动画实现

项目使用 Framer Motion 实现：

- 页面切换过渡
- 弹窗出现与消失
- 底部面板上滑
- 下拉菜单展开与收起

这些动画主要用于增强原型演示时的完整度和交互感。

## 9. 构建流程说明

### 9.1 类型检查

执行命令：

```bash
npm run check
```

实际逻辑：

- 调用 TypeScript 编译器
- 对项目源码做类型校验
- 不输出运行产物

### 9.2 构建命令

执行命令：

```bash
npm run build
```

实际逻辑：

1. 生成 `dist/styles.css`
2. 将 `src/` 下的 TypeScript / TSX 编译到 `dist/`
3. 校验关键文件是否存在

### 9.3 预览命令

执行命令：

```bash
npm run dev
```

实际逻辑：

1. 先执行一次构建
2. 启动本地静态服务
3. 默认监听 `127.0.0.1:4173`
4. 如果端口被占用，会自动顺延

## 10. 部署说明

### 10.1 当前版本适合的部署方式

当前版本最适合：

- 本地演示
- 课程作业展示
- 内网静态部署
- 原型评审环境

### 10.2 构建产物说明

执行 `npm run build` 后，主要产物在 `dist/` 目录下，包括：

- `app.js`
- `main.js`
- `styles.css`
- 编译后的 `components/`
- 编译后的 `data/`
- 编译后的 `vendor/`

### 10.3 部署时必须保留的目录

如果要把项目部署到其他机器或服务器，以下内容必须一起保留，且相对路径不能改乱：

- `index.html`
- `dist/`
- `reference/`
- `node_modules/react/`
- `node_modules/react-dom/`
- `node_modules/framer-motion/`

原因：

- 页面入口在 `index.html`
- 样式和业务代码在 `dist/`
- 部分图片素材在 `reference/`
- 运行时依赖从 `node_modules` 的 UMD 文件直接加载

### 10.4 推荐部署步骤

#### 方案一：本地机器演示

1. 拷贝整个项目目录
2. 执行 `npm install`
3. 执行 `npm run build`
4. 执行 `npm run dev`
5. 浏览器访问终端输出地址

这是当前版本最稳妥的方式。

#### 方案二：静态服务器部署

前提：

- 服务器能够按当前目录结构提供静态文件
- 允许访问 `node_modules` 中被引用的运行时脚本

步骤：

1. 在服务器环境安装依赖
2. 执行 `npm run build`
3. 使用静态服务器指向项目根目录
4. 确保 `index.html` 能正确访问 `dist/`、`reference/` 和 `node_modules/`

### 10.5 部署注意事项

- 不建议只上传 `dist/` 目录，因为当前版本不是完全自包含构建
- 不建议直接删掉 `reference/`，否则部分页面素材会丢失
- 如果静态服务器禁止访问 `node_modules/`，当前版本会无法正常运行

## 11. 已知特点与后续优化建议

### 11.1 当前版本特点

- 更偏向高保真交互演示
- 页面文案和候选数据高度配置化
- 三条业务线共用一套页面逻辑

### 11.2 后续可优化方向

1. 将当前部署方式改为标准 `vite build` 自包含产物
2. 把运行时依赖从 `node_modules` 引用改为打包进最终输出
3. 补充更细的异常态和空状态
4. 增加真实接口模拟或 Mock 数据层
5. 将部分页面组件进一步拆分，提高维护性

## 12. 总结

本项目已经完成了从高保真原型到前端交互 Demo 的核心落地，适合用于：

- 课程作业提交
- 原型演示
- 产品评审
- 后续继续迭代为正式前端项目的基础版本

如果后续继续开发，建议优先推进两件事：

2. 继续细化页面状态与真实业务逻辑
